# Split-contract diagram and conformance report — 2026-06-29

## Scope

This report covers the reduction from a monolithic v0 Solidity settlement adapter to:

- `contracts/ARKBirthCertificate.sol` — ERC-721-compatible agent birth certificate.
- `contracts/ARKRuntimeMarketplace.sol` — compatible runtime marketplace / settlement adapter.

It also covers README Mermaid diagram conformance and the local deployment receipt generated by `scripts/deploy.cjs`.

## Reduction summary

- Birth certificates are now standard ERC-721 tokens rather than custom token bookkeeping embedded in the runtime marketplace.
- Runtime marketplace authorization depends only on `ownerOfAgent(agentId)` compatibility.
- Runtime state was removed from the birth certificate record; `activeClaimByAgent` is marketplace state.
- Separate release-receipt and state-root operations were merged into one `checkpointState` operation.
- Direct HostOffer claiming, dedicated payment escrow, attestation archive, service bundles, and dispute registry methods remain provider-neutral Reality Ledger primitives / roots rather than v0 marketplace methods.

## Requirement-relaxation notes

No feature reduction was required for this reduction. The key simplification was interpreting the broad Reality Ledger primitives as provider-neutral roots/receipts unless the MVP needs a dedicated Solidity method.

Requirements that would meaningfully increase v0 contract surface if made mandatory on-chain now:

1. Direct user claiming of pre-warmed `HostOffer` objects.
2. Dedicated on-chain `PaymentEscrow` payout/refund accounting.
3. Dedicated on-chain `AttestationArchive` and `DisputeRegistry` methods.
4. On-chain `ServiceOffer` / `ServiceReceipt` bundles.
5. On-chain slashing for non-cryptographic uptime/performance claims.

Keeping those as ledger roots or later adapters preserves the feature-file requirements while keeping the v0 contract surface small.
